# 实时快照与模型任务审计说明

## 目标

本次改造解决两个独立问题：

1. 比赛赔率持续变化时，旧 AI 报告不能被误当作当前盘口结论。
2. 模型服务和走势抓取发生异常时，必须保留可复核的输入、可见输出、阶段和失败原因。

## 数据流

### 单场分析

1. 前端发起 `POST /api/match_ai_analysis`，分析请求使用 `force: true`。
2. 服务端抓取当前比赛详情和赔率索引。
3. 服务端刷新每个可用公司/市场的让球与大小球历史走势。
4. 只有详情校验和走势校验通过后，才构建本次 AI 上下文。
5. 同一快照并发生成三份分析报告，再由 CRO 汇总为最终报告。
6. 保存报告缓存、预测记录和分析 trace。

### 批量分析

1. 批量任务先串行准备每一场的详情和走势，避免把同一上游放大为突发并发请求。
2. 数据准备完成后，模型请求共享全局并发槽位，最多三条流同时运行。
3. 每场保留独立任务键、快照 hash、报告缓存和 trace。

## 快照与缓存规则

- `details_<match_id>.json` 是详情抓取缓存，不等于 AI 报告缓存。
- AI 报告缓存版本为 `7`。
- 赛前报告只有在快照采集后 90 秒内才可复用。
- 滚球报告永不复用；每次分析均需使用新快照。
- 赔率索引会按公司 ID 和名称排序后计算 `market_snapshot_hash`，避免公司返回顺序变化造成误失效。
- 详情刷新得到新的盘口 hash 时，关联的 `ai_analysis_<match_id>.json` 会被删除。
- 旧版本、缺少快照时间、缺少盘口 hash、过期或不完整的报告缓存均不可展示。

## 模型执行策略

- 三路子分析总时限：300 秒。
- CRO 汇总总时限：240 秒。
- 连接超时：30 秒；流读取超时：120 秒。
- 全局模型流并发：3。
- 没有收到任何可见报告文本时，连接/读取/TLS 临时错误会重试一次。
- 已收到可见文本后不重复请求，避免产生重复报告或覆盖已获得内容。
- 批量单场看门狗按队列模型设置为 5400 秒。

## 审计记录

模型 trace 位于 `cache/analysis_traces/`，包含：

- `analysis_input`：本次快照生成的上下文。
- `analyst_inputs`：三路请求消息和温度参数。
- `analyst_outputs` 与 `analyst_reports`：每路可见输出、状态、首个流事件时间和错误。
- `cro_input`、`cro_output`、`final_report`：最终汇总的输入和可见输出。
- `snapshot_hash`、模型名称、任务阶段、开始/结束时间和错误。

trace 不保存 API Key、Authorization 请求头或原始推理串，只保存可见报告和可审计元数据。

## 走势数据门禁

走势刷新会针对详情快照中标记为可用的市场请求数据。无可用市场的公司不会被请求。

系统现已支持**部分走势降级分析 (Degraded Trend Analysis)** 策略：
- 当走势获取覆盖率 ≥ 55%（且至少获取 1 家公司变盘历史）时，允许进入 AI 研判阶段。
- 系统会在上下文、快照质量及审计 Trace 中明确标记 `degraded: true`，注明当前走势覆盖率（如 `13/22`），并进行置信度降级提示。
- 若覆盖率低于 55%，依然触发门禁阻断，防止伪造完整行情。

挑战页会被明确识别为 `WAF challenge response persisted`，并包含二次 Node 解密重试与自动自愈机制。

## 模型报告硬一致性校验

首席风险官 (CRO) 汇总前引入了 `validate_report_recommendation_consistency` 硬校验逻辑：
- 严格审计精算师报告中“变盘真伪解码”提及的优势方（如“真实期望值在客队”）与“亚洲让球盘推荐”切入方向是否对立。
- 一旦检测到方向矛盾或盘口倒置，标示该子报告硬校验失败，确保模型报告推荐逻辑与证据 100% 严密自洽。

## 准备阶段审计 Trace

当比赛处于详情拉取或变盘历史同步阶段发生阻断/失败时，系统将通过 `_persist_preparation_failure_trace` 自动持久化生成独立的准备阶段审计日志（`cache/analysis_traces/{match_id}_prep_failed_*.json`），完整保留当时的错误类型与行情质量快照。
